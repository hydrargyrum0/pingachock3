# Слияние pingachock-2.0 (Telegram-бот) в pingachock3

## Контекст

`https://github.com/hydrargyrum0/pingachock-2.0` — отдельный репозиторий, Telegram-бот
(TypeScript/Node, `telegraf`), который раньше ходил в другой бекенд ("Router Ping API 2.1.0",
donor-репо содержит его OpenAPI-спеку `astroping-openapi.json`). Бот перестаёт быть привязан к
старому API и переезжает на API `pingachock3`. Это полный перенос: ping/health-отчёты,
Remnawave-интеграция, Vultr-интеграция, админ-панель управления доступом — всё остаётся.

Донорский бот — не просто "бот для пинга", а бот мониторинга инфраструктуры с четырьмя частями:

1. **Core ping/health** — ручной пинг через меню, health-report по кастомному списку целей.
2. **Remnawave-интеграция** — тянет список хостов из VPN-панели Remnawave, пингует их.
3. **Vultr-интеграция** — тянет VPS-инстансы по тегу из Vultr Cloud API, пингует их.
4. **Админка/доступ** — whitelist телеграм-пользователей, управление узлами старого API,
   рассылки.

Плюс планировщик: раз в N секунд гоняет health-check на пользователя, шлёт в чат только проблемы.

## Ключевое архитектурное отличие от старого API

Старый бекенд держал с "роутерами" **постоянный WebSocket** (`/ws`, роутер подключается с
токеном) — поэтому `GET /api/ping` был синхронным: бекенд слал команду в открытый сокет и ждал
ответ в рамках того же HTTP-запроса.

`pingachock3` использует **поллинг**: узел стучится на `/agent/poll` раз в ~30с и забирает
задания. Это осознанный выбор ради stateless-масштабирования (см. `docs/ARCHITECTURE.md`) и
в этом проекте не меняется. Следствие: check через узел занимает от создания до результата
порядка 20-40 секунд, а не доли секунды.

## Решения

### 1. Sync/async — гибрид

- **Пинг с сервера** (`router_name="server"` в терминах старого API) — синхронный. Бекенд не
  зависит ни от какого узла, значит может выполнить проверку сам и отдать результат в том же
  HTTP-ответе.
- **Пинг через узел** — остаётся асинхронным (создать check → поллить `GET /checks/{id}` до
  `done`). Это ок для периодических health-report'ов (не чувствительны к задержке в десятки
  секунд) и приемлемо для интерактивного "🔍 Ping" в чате.
- Архитектура бекенда (poll-модель, stateless) не меняется. WebSocket для узлов не вводим —
  это был бы отдельный большой пересмотр, не в рамках этой задачи.

### 2. Новый эндпоинт: `POST /api/v1/server-ping`

Отдельный эндпоинт, **не** через `checks`/`check_runs` — ничего не пишет в БД, выполняется
синхронно в своей горутине на запрос.

```
POST /api/v1/server-ping
Auth: Authorization: Bearer <api_key>  (обычный api_key, без admin-scope)

Request:
{
  "targets": ["1.1.1.1", "ferul.grokotun.ru"],
  "ports": ["icmp", "80", "443"]   // опционально, default ["icmp"]
}

Response 200:
{
  "results": [
    {
      "target": "1.1.1.1",
      "resolved_ip": "1.1.1.1",
      "icmp": {"success": true, "latency_ms": 4},
      "ports": {"80": "open", "443": "open"}
    },
    {
      "target": "ferul.grokotun.ru",
      "resolved_ip": "185.253.75.61",
      "icmp": {"success": true, "latency_ms": 6},
      "ports": {"80": "closed"}
    }
  ]
}
```

Реализация переиспользует существующий `internal/checks` (`PingChecker`/`TCPChecker`) с пустым
`NetConfig{}` (системный маршрут по умолчанию сервера, не привязан ни к какому интерфейсу узла).

**Ограничения:** максимум целей за один запрос и таймаут на весь запрос — чтобы один батч не
положил бекенд (конкретные числа — на этапе реализации, ориентир: сопоставимо с текущим
`POLL_BATCH_LIMIT`).

**Требование корректности (жёсткое, из реального прошлого инцидента):** в старом API были
случаи, когда ответ на пинг одного пользователя (`1.1.1.1`) приходил другому, ожидавшему свой
пинг (`1.2.3.4`) — похоже на глобальное состояние вместо per-request корреляции в
WebSocket-реализации. У бота будет до 10 одновременных пользователей. `server-ping` обязан
исполнять каждый запрос независимо — без общих переменных/очереди между конкурентными
запросами; каждый HTTP-запрос отвечает сам за себя в своей горутине. Это следует из обычной
модели `net/http` в Go при условии, что реализация не вводит расшаренное состояние сама. Указать
явный тест на конкурентность (N одновременных запросов на разные цели → каждый ответ содержит
именно свою цель) как часть реализации, не полагаться на "и так должно работать".

### 3. ICMP + порты одним запросом — оркестрация, без изменения схемы

Старый API отдавал ICMP и несколько TCP-портов одним ответом на цель. Схема `checks`
(`check.type` — один тип: `ping`/`tcp`/`http`/`dns`) не меняется.

Это касается только **пинга через узел**: интеграционный слой бота создаёт 1 ping-check (batch
по targets) + по одному tcp-check на каждый запрошенный порт, дожидается всех, склеивает в один
плоский объект на цель для рендера в чате. **`server-ping` (раздел 2) комбинирует ICMP+порты
нативно в одном запросе** — он не завязан на схему `checks` вообще, так что оркестрация ему не
нужна.

### 4. Авторизация бота

Один `api_key` на весь бот (аккаунт "telegram-bot" через существующий admin-flow
`POST /accounts` + `POST /accounts/{id}/api-keys`), хранится в настройках бота вместо старой
пары `admin_token`+`client_token`. Whitelist телеграм-пользователей остаётся собственным
(nedb, `userRepo`) — не завязан на api_key/account модель pingachock, это её нормальное
использование (один api_key = одно внешнее приложение).

**Миграция данных бота:** `data/users.db` (whitelist телеграм-пользователей) переносится как
есть — формат не меняется. В `data/settings.db` меняют смысл только ключи `admin_token`/
`client_token` (заменяются на один `api_key` + `api_url` уже указывает на pingachock, не на
старый бекенд) — остальные (`health_report:*`, `health_periodic:*`, `remnawave:*`, `vultr:*`)
переносятся без изменений, их формат от смены бекенда не зависит.

### 5. Небольшие дополнения к `nodes`

Старая админка умела:

- **Блокировать** роутер, не удаляя (сохраняет историю, временно исключает из диспетчеризации).
  → `nodes.blocked bool default false` + `PUT /api/v1/nodes/{id}` (`{"blocked": true}`,
  admin-scope) + `dispatch.Resolve` пропускает заблокированные узлы.
- Показывать **platform** узла. Агент уже шлёт `agent_version` в `POST /agent/poll` — добавить
  туда же `platform: runtime.GOOS` (`windows`/`linux`/`darwin`), копеечное изменение по тому же
  паттерну, что уже есть.

### 6. Язык бота — остаётся TypeScript/Node

`telegraf` — зрелый, проверенный фреймворк, вся UI/меню/session-логика (~3000 строк
`index.ts`) уже работает и не завязана на конкретный бекенд-API. Переписывается только
`api-client.ts`. Репозиторий становится полиглот-монорепо (Go бекенд + Node бот) — как
docker-compose сервис рядом, это нормально, не проблема.

**Важный побочный эффект решения по (3) сохранить плоский формат результата**: раз
интеграционный слой сам приводит ответ pingachock к тому же плоскому виду
(`{ip, resolved_ip, status, ICMP, port_80, ...}`), который уже понимают
`formatPingResultLine`/`summarizePingResponse` и остальной рендеринг — **UI/меню-код бота
почти не трогается**. Меняется по сути только `api-client.ts`.

### 7. Структура репозитория и деплой

- Код бота → `bot/` в корне `pingachock3`, рядом с `cmd/`, `internal/`, `migrations/`.
- Новый сервис `bot` в `docker-compose.prod.yml`: `depends_on: backend`, сеть `internal` (та же,
  что у `postgres`/`backend`/`caddy` уже сейчас), `API_URL=http://backend:8080` — то есть в
  обход Caddy и порта 30031, порт наружу не публикуется вообще (Telegram long-polling, входящих
  подключений к боту не требуется).
- `bot/data/` (nedb: `users.db`, `settings.db`) — отдельный volume, по аналогии с `pgdata`.
- Старый репозиторий `pingachock-2.0` просто перестаёт использоваться после переноса.

### 8. Что не меняется

- Remnawave/Vultr-интеграции — как есть, ходят в сторонние API напрямую (`fetch` к Remnawave
  panel / Vultr Cloud API), завязаны на pingachock только в финальном вызове пинга
  извлечённых целей.
- Whitelist телеграм-пользователей (`userRepo`, nedb) и вся админ-панель бота (кроме
  "API URL"/токен, которые сворачиваются в одну пару вместо `admin_token`+`client_token`).
- Периодический health-scheduler — механика (тик каждые 5с, интервал на пользователя) та же,
  меняется только то, какой клиент он вызывает внутри.

## Открытые вопросы для этапа реализации (не блокируют дизайн)

- Точные лимиты `server-ping` (макс. `targets` за запрос, таймаут).
- "auto" в семантике роутера бота → маппится на "взять первый `online` узел из `GET /nodes`"
  (простой выбор, не round-robin) — подтвердить на этапе реализации, если понадобится другая
  логика.
- Судьба старого репозитория `pingachock-2.0` (архивировать/оставить как есть) — не влияет
  на реализацию, чисто организационный момент.
